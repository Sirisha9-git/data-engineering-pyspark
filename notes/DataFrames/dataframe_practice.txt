# DIFFERENT WAYS TO CREATE PYSPARK DATAFRAME

1. Create a PySpark DataFrame from a list of rows
✔ When to use it
Quick testing
Small sample data
Mocking interview problems
No schema needed (Spark infers it)
✔ What happens
Spark looks at the Python data types
Automatically infers the schema
Creates a DataFrame
✔ Why this is useful
Fastest way to create a DataFrame for practice or debugging.

2. Create a PySpark DataFrame with an explicit schema
✔ When to use it
You want full control over data types
You want to avoid schema inference
You need strict types for ETL pipelines
You want to enforce nullable / non‑nullable fields
You’re working with production‑like data
✔ What happens
Spark uses your schema, not inferred types
You control:
column names
data types
nullability
✔ Why this is useful
This is how real data engineering pipelines are built.

3. Create a PySpark DataFrame from a pandas DataFrame
✔ When to use it
You already have data in pandas
You’re converting local data → Spark
You’re doing hybrid pandas + PySpark work
You want to scale pandas code to Spark
✔ What happens
Pandas DataFrame is converted to Spark DataFrame
Spark infers schema from pandas types
Useful for small → medium datasets
✔ Why this is useful
Great for:
Data exploration
Converting CSV/Excel data
Migrating pandas code to Spark

------------------------------------------------------------------------------
# create a PySpark DataFrame from a list of rows

# way1
from datetime import datetime, date
from pyspark.sql import Row

df1 = spark.createDataFrame([
    Row(id=1,name = 'john',dob=(date(1990,1,1,)),salary=10000),
    Row(id=2,name='mary',dob=date(1995,3,11),salary=5000)
])
df1.show()

Execution time : 482 ms

This uses Row objects.
Row objects require extra processing.
Spark must:
1. Inspect each Row object
2. Extract field names
3. Extract field values
4. Infer schema from Row metadata (Schema inference is expensive).
5. Convert Row → Internal Spark Row format
This adds overhead.
When to use Row objects->
Rarely — only when you need named fields inside Python before creating the DataFrame.
NOT recommended for production.


------------------------------------------------------------------------------

# create a PySpark DataFrame from a list of rows

# way2 - high performance comapred to way1
# need not import Row class
data = [(1,'john',date(1990,1,1),10000),(2,'mary',date(1995,3,11),5000)]
df2 = spark.createDataFrame(data,["id","name","dob","salary"])
df2.show()

Execution time : 191 ms

This uses Python tuple object.
Spark:
Reads the tuple
Uses the column names you provided
Infers schema directly from Python types
Converts tuple → Internal Spark Row format
No metadata extraction.
No Row parsing.
No field-name lookup.
This is much faster.
Spark skips schema inference for field names since field names are already given.(skips the expensive part)
Recommended for production.

# Create a PySpark DataFrame with an explicit schema.

# Create pyspark dataframe with an explicit schema
from datetime import date,datetime
from pyspark.sql.types import StructType,StructField,LongType,DoubleType,StringType,DateType,TimestampType

schema = StructType([
    StructField("a",LongType(),True),
    StructField("b",DoubleType(),True),
    StructField("c",StringType(),False),
    StructField("d",DateType(),True),
    StructField("e",TimestampType(),False)
    ])
df3 = spark.createDataFrame([(1,2.0,"str1",date(1990,1,1),datetime(2000,4,13,5,10,10)),
                             (5,6.0,"str2",date(1995,2,5),datetime(1997,6,10,10,10,15))
                             ], schema = schema
                            )
df3.show()
# Execution time : 458ms

imports ->
✔ What this does
date, datetime → Python date/time objects  - for python tuples.
StructType, StructField → Used to define a schema
LongType, DoubleType, StringType, DateType, TimestampType → Spark data types
✔ Why this matters
Spark needs Spark-specific data types to build a schema.
Python types alone are not enough.

Defining Schema ->
✔ What this does
You are defining the structure of the DataFrame manually.
StructType([...])
Represents the entire schema (like a table definition).
StructField("a", LongType(), True)
Each StructField defines one column
repeat this for all columns.
✔ Why this matters
You avoid schema inference
You enforce strict data types
You control nullability
This is production‑grade ETL style

Creating the DataFrame ->
✔ What this does
You pass a list of Python tuples
You pass your explicit schema
Spark converts the tuples → internal Spark rows using your schema
✔ Why this is the best performance method
Tuples are lightweight
No Row object overhead
No schema inference
Spark directly applies your schema
Fastest way to create a DataFrame manually

# Creating a pyspark dataframe from pandas dataframe

1. Load or create data in pandas, which lives in local Python memory (CSV/Excel/API/manual creation).
2. Pass the pandas DataFrame to Spark using spark.createDataFrame(pdf), where Spark converts local rows into distributed Spark rows.
3. Spark distributes the data across the cluster, applies schema inference (unless you provide one), and creates a fully parallelized Spark DataFrame ready for big‑data processing.

# Collecting Data to Driver

1. Collect()
    - brings all data to the driver.
    - If the dataset is large → OutOfMemoryError.
    - Use only for:
        Small datasets
        Debugging
        Unit tests
Alternatives to collect to avoid out of memory issues.

2. take  /  tail
    - df.take(N)  : Take first N rows
    - df.tail(N)  : Take last N rows
        ✔ Much safer than collect()  
        ✔ Does NOT load entire dataset
        ✔ Works well for sampling

# Selecting and Accessing Dataframe data

1. Lazy Evaluation
    PySpark DataFrames are lazily evaluated — no computation runs until an action is called.
    Selecting a column (df.a) returns a Column object, not data.
    Column-wise operations like upper(), isNull() also return Column objects.
    Computation triggers only on actions: .show(), .collect(), .count(), .write.

2. Column References
    Dot notation: df.a
    Bracket notation: df["a"] — preferred when column names have spaces or special characters.
    col() function: from pyspark.sql.functions import col → col("a").
    All three return the same Column object.

3. select() — Pick Columns
    Returns a new DataFrame with only the chosen columns.
    Accepts one or more Column objects or column name strings.
    Original DataFrame remains unchanged (immutable).
    SQL equivalent: SELECT c FROM df.

4. withColumn() — Add or Replace a Column
    Takes two arguments: column name (string) and column expression.
    If the name exists → overwrites that column.
    If the name is new → appends the column.
    Returns a new DataFrame; original is unchanged.
    SQL equivalent: SELECT *, UPPER(c) AS upper_c FROM df.

5. filter() / where() — Pick Rows
    Both are aliases — identical behavior.
    Accepts a Column condition that evaluates to boolean.
    Returns a new DataFrame with matching rows only.
    Supports standard operators: ==, !=, >, <, >=, <=.
    Chain multiple filters: df.filter(cond1).filter(cond2).
    SQL equivalent: WHERE a = 1.

6. Transformations vs Actions
    Transformations (lazy): select(), filter(), withColumn(), groupBy(), join() — build the execution plan only.
    Actions (trigger execution): show(), collect(), count(), first(), take(n), write.
    Spark optimizes the full chain of transformations before running anything — this is the Catalyst optimizer.

7. Common Pitfalls
    Forgetting .show() and assuming select() printed results.
    Using Python == in filter() is correct; using = is assignment and will error.
    withColumn() does not modify in place — always assign the result to a variable.
    Column name conflicts after join() — use alias() or bracket notation to disambiguate.
